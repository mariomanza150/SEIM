# SEIM Project Structure Guide

**Last Updated:** 2025-12-05  
**Purpose:** Comprehensive guide to project directory structure and organization

---

## Overview

SEIM follows a clear organizational structure to separate concerns and make the codebase maintainable:

- **Application Code** - Django apps and core functionality
- **Documentation** - Separated into generated and manual documentation
- **Configuration** - Environment, Docker, and tool configurations
- **Tests** - Comprehensive test suite
- **Tools** - AI agent configurations and IDE settings

---

## Documentation Structure

### `docs/` - Generated/Auto-Generated Documentation

**Purpose:** Documentation that is automatically generated or updated by tools.

**Contents:**
- `architecture.md` - Auto-generated system architecture
- `api-contracts.md` - Auto-generated API documentation
- `data-models.md` - Auto-generated database schema
- `component-inventory.md` - Auto-generated component catalog
- `source-tree-analysis.md` - Auto-generated directory analysis
- `project-overview.md` - Auto-generated project summary
- `development-guide.md` - Auto-generated development guide
- `index.md` - Index of generated documentation

**Subdirectories:**
- `docs/status/` - Status reports and progress tracking
- `docs/guides/` - Quick reference guides
- `docs/analysis/` - Analysis documents

**When to Use:**
- Reference for understanding current system state
- Quick lookup for API contracts and data models
- Understanding component structure

**Maintenance:**
- Regenerated by tools (BMAD document-project workflow, etc.)
- Do not manually edit (changes will be overwritten)
- If updates needed, modify source code and regenerate

### `documentation/` - Manual/Maintained Documentation

**Purpose:** Documentation that is manually written and maintained by the team.

**Contents:**
- `README.md` - Documentation index
- `architecture.md` - Manual architecture documentation
- `developer_guide.md` - Developer guide
- `api_documentation.md` - Manual API documentation
- `installation.md` - Installation instructions
- `deployment.md` - Deployment guide
- `testing.md` - Testing guide
- `business_rules.md` - Business logic documentation
- `roadmap.md` - Development roadmap
- `backlog.md` - Current tasks
- `user_stories.md` - User requirements

**Subdirectories:**
- `documentation/archive/` - Historical documentation
- `documentation/audit_reports/` - Audit reports
- `documentation/implementation_plans/` - Implementation plans
- `documentation/wireframes/` - UI/UX wireframes
- `documentation/sphinx/` - Sphinx documentation source

**When to Use:**
- Authoritative source for how to use the system
- Development guidelines and best practices
- Planning and roadmap information
- User guides and tutorials

**Maintenance:**
- Manually updated by developers
- Should be kept up-to-date with code changes
- Reviewed during code reviews

## Tool Configuration Structure

### `.tools/` - AI Agent and IDE Configurations

**Purpose:** Consolidates all tool-specific configuration directories.

**Structure:**
```
.tools/
├── agents/              # AI agent configurations
│   ├── .agent/
│   ├── .agentvibes/
│   ├── .augment/
│   ├── .bmad/          # BMAD Method workflows and agents
│   ├── .claude/        # Claude AI configurations
│   ├── .clinerules/
│   ├── .codex/
│   ├── .crush/
│   ├── .gemini/
│   ├── .iflow/
│   ├── .opencode/
│   ├── .qwen/
│   ├── .rovodev/
│   └── .trae/
└── ide-config/         # IDE-specific configurations
    ├── .cursor/        # Cursor IDE settings
    └── .windsurf/      # Windsurf IDE settings
```

**Purpose:**
- Keeps root directory clean
- Makes it clear these are tool configurations
- Easier to manage and document
- Can be excluded from certain operations

**Note:** These directories are typically hidden (start with `.`) and contain tool-specific settings. They don't affect application functionality.

## Application Code Structure

### Django Apps

Each Django app follows a standard structure:

```
app_name/
├── __init__.py
├── admin.py           # Django admin configuration
├── apps.py            # App configuration
├── models.py          # Database models
├── views.py           # View functions/classes
├── serializers.py     # DRF serializers
├── services.py        # Business logic (service layer)
├── urls.py            # URL routing
├── filters.py         # Query filters (if applicable)
├── tasks.py           # Celery tasks (if applicable)
├── migrations/        # Database migrations
├── tests/             # App-specific tests
└── management/        # Management commands (if applicable)
    └── commands/
```

### Key Apps

- **`accounts/`** - User management, authentication, profiles
- **`exchange/`** - Exchange programs and applications
- **`documents/`** - Document upload and validation
- **`notifications/`** - Email and in-app notifications
- **`analytics/`** - Reporting and metrics
- **`grades/`** - Grade translation system
- **`cms/`** - Wagtail CMS integration
- **`core/`** - Shared utilities, permissions, caching

## Best Practices

### Adding New Files

1. **Documentation:**
   - Generated docs → `docs/`
   - Manual docs → `documentation/`
   - Status/Progress → `docs/status/`
   - Guides → `docs/guides/` or `documentation/`

2. **Configuration:**
   - Tool configs → `.tools/agents/` or `.tools/ide-config/`
   - Environment → `.env` (not committed)
   - Docker → Root level with descriptive names

3. **Code:**
   - Follow Django app structure
   - Place in appropriate app directory
   - Add tests in `tests/` directory

### Finding Documentation

- **Need authoritative info?** → Check `documentation/`
- **Need current system state?** → Check `docs/`
- **Need quick reference?** → Check `docs/guides/` or `documentation/`
- **Need status/progress?** → Check `docs/status/`

### Maintenance

- Keep root directory clean (< 10 visible files)
- Consolidate tool configurations
- Separate generated vs manual documentation
- Update this guide when structure changes
